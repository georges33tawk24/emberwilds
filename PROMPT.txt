THE QUALITY BAR (non-negotiable — apply to your own work before showing it)
Judge everything by: "Would this feel complete if it shipped in a commercial
game? Does it look polished? Does it behave naturally? Are there any missing
animations, VFX, interactions, transitions, or feedback?" Concretely:
- Handcrafted, not generated-looking: intentional pacing, no flat repetitive
  corridors.
- Consistent art language: obey the "No-Neon Bible" palette in src/gfx/palette.ts
  (warm/desaturated; banned: electric cyan, hot magenta, acid green, LED glow).
- Juice on every action: screenshake, hitstop, squash/stretch, particles, sound,
  camera work, transitions. A static response to any action is a bug.
- Mobile is first-class: full-screen fill on every aspect ratio, no black bars,
  thumb-friendly controls, every on-screen element useful and consistent.
- Water/environment must feel physical: animated surfaces, objects INSIDE water,
  splashes/ripples on entry, submerged tinting, believable depth & render order.
- Proactively find and fix rough edges; continuously improve earlier work; ship
  every feature production-ready. Cost/effort is not the constraint — quality is.

DEFINITION OF DONE (every change):

4) VERIFIED in the browser (run it, screenshot it — never claim an unobserved

KEY PATTERNS YOU MUST RESPECT (full detail in HANDOFF.md):
- Deterministic 120 Hz sim + render interpolation; sim classes (src/entities/*)
  are PURE and Phaser-free — no Date.now()/Math.random() in sim logic.
- Pixel art is authored AS CODE: string arrays of palette codes baked at boot;
  every row in a frame-group must be equal width (validate before baking).
- Levels are ASCII grids (src/data/levels/*.ts). tests/levelLint.test.ts
  auto-tests every level for reachability — only GOAL and CHECKPOINT must be
  reachable, so tokens/secrets can be skill-gated. Author with a Node script whose
  validate() mirrors the lint, iterate to OK, then write the .ts.
- Water is REGIONS (LevelDef.water rects -> waterSet), decoupled from tiles.
- Bosses are config-driven (BOSS_CONFIGS in src/entities/bossSim.ts).
- Responsive: VIEW.w = 360 x aspect (src/gfx/viewport.ts), read at RUNTIME by
  every scene; Scale.FIT fills any landscape aspect; portrait -> rotate overlay.
- NEVER use pointer events for touch controls over the Phaser canvas (iOS Safari
  suppresses them -> dead taps). Use real touch events + global hit-test.
